# Backlog Issue キー自動検出

ユーザーのメッセージに Backlog の課題キー（例: `PROJ-123`, `API_V2-45`）が
含まれていたら、明示的な指示がなくても、対象 Issue の基本情報を静かに取得し
会話の文脈を補強する。

## 検出パターン

正規表現（概念）：`[A-Z][A-Z0-9_]+-\d+`

該当する例：`PROJ-1`、`FOO_BAR-321`、`ILU-99`

該当しない例：
- `UTF-8`、`GPT-4`（先頭が1文字、または一般用語として流通）→ 誤検出注意
- 数字-数字（`2024-10`）や単なる範囲（`1-5`）
- 日付や電話番号のようにハイフン周辺が数字のみ

曖昧な場合は検出しないか、確信度を下げて文脈で判断する。日本語文中では
「ILU-123について」のように助詞と隣接することが多いので、
前後のコンテキストで Backlog 課題の言及かを見極める。

## トリガー時の動作

1. メッセージから候補キーを抽出（重複は排除）
2. 同じキーが**直近の会話でまだ参照されていない場合のみ** `mcp__backlog__get_issue`
   を呼ぶ。既に取得済みなら再取得しない
3. 取得できたら以下を会話に自然に織り込む：
   - 件名
   - ステータス
   - 担当者（いれば）
   - 期限（あれば）
4. 取得に失敗した場合（存在しない、権限なし等）は、**黙って無視**する。
   ユーザーの質問に直接関係ない可能性が高いため、エラーメッセージを前面に出
   さない。ただしユーザーがその課題について明示的に質問している場合はエラー
   理由を説明する

## 発話への織り込み方

- ユーザーの質問がその課題に関するものなら、情報を見出し付きで整形して回答
  の一部とする
- そうでなく単に言及しただけなら、末尾に控えめに補足（例：「※ `PROJ-123`
  は『ログイン不具合』/ 進行中 / 担当: 山田」）
- 補足は簡潔に、1〜2行に収める

## 過剰反応を避ける

- 1通のメッセージに同じキーが複数回出てきても1回だけ取得
- キーが10個以上ある場合は、ユーザーに「どれを詳しく見ますか」と確認
- ユーザーがキーを**コードブロック内で提示している**（例：ログ貼付）場合は
  自動取得しない。会話の主題ではない可能性が高い

## プロジェクトキーの推定

検出されたキーのプレフィックス（`PROJ` 部分）は、以後の会話で
プロジェクトを省略された場合のデフォルトとして記憶しておく。ただしユーザー
が別プロジェクトを明示したらそちらを優先する。

## 参考ツール

- `mcp__backlog__get_issue` — キーで直接取得可能
- `mcp__backlog__get_issue_comments` — コメントまで踏み込む必要があるとき
